The Jobs API allows you to monitor and manage background jobs for indexing, syncing, and find-and-replace operations.

## List Jobs

Get all background jobs with optional filtering and pagination.

```http
GET /api/v1/jobs
```

### Query Parameters

| Parameter | Type    | Description                                    |
| --------- | ------- | ---------------------------------------------- |
| `type`    | string  | Filter by job type (`index`, `sync`, `replace`)|
| `status`  | string  | Filter by status                               |
| `exclude_status` | string | Exclude jobs with this status (e.g., `completed`) |
| `repo`    | string  | Filter by repository name (partial match)      |
| `limit`   | integer | Results per page (default: 50, max: 500)       |
| `offset`  | integer | Pagination offset                              |

### Response

## Cancel Job

Cancel a pending job. Only jobs with status `pending` can be cancelled.

```http
POST /api/v1/jobs/{id}/cancel
```

### Response

```json
{
  "status": "cancelled",
  "message": "Job cancelled"
}
```

<Aside type="caution">
  Only pending jobs can be cancelled. Running jobs cannot be interrupted.
</Aside>

## Cleanup Jobs

Remove old completed and failed jobs. This helps prevent Redis from growing indefinitely.

```http
POST /api/v1/jobs/cleanup
```

### Query Parameters

| Parameter | Type    | Description                                    |
| --------- | ------- | ---------------------------------------------- |
| `max_age_hours` | integer | Maximum age in hours for jobs to keep (default: 1) |

### Response

```json
{
  "status": "ok",
  "deleted_count": 42,
  "scanned_count": 150,
  "max_age_hours": 1,
  "message": "Cleanup completed"
}
```

<Aside type="tip">
  The scheduler automatically runs cleanup every 10 minutes based on the `job_retention` setting.
  Use this endpoint for manual cleanup or to apply a different retention period.
</Aside>

## Bulk Cancel Jobs

Cancel all pending and/or running jobs matching the given filters. Only pending and running jobs can be cancelled.

```http
POST /api/v1/jobs/cancel-all
```

### Query Parameters

| Parameter | Type    | Description                                    |
| --------- | ------- | ---------------------------------------------- |
| `type`    | string  | Filter by job type (`index`, `sync`, `replace`)|
| `status`  | string  | Filter by status (`pending`, `running`)        |

### Response

```json
{
  "status": "ok",
  "processed": 25,
  "succeeded": 24,
  "failed": 1,
  "message": "Bulk operation completed"
}
```

<Aside type="caution">
  Cancelled jobs are marked as `failed` with an error message indicating they were cancelled.
</Aside>

## Bulk Delete Jobs

Delete all completed and/or failed jobs matching the given filters. Only completed and failed jobs can be deleted.

```http
POST /api/v1/jobs/delete-all
```

### Query Parameters

| Parameter | Type    | Description                                    |
| --------- | ------- | ---------------------------------------------- |
| `type`    | string  | Filter by job type (`index`, `sync`, `replace`)|
| `status`  | string  | Filter by status (`completed`, `failed`)       |

### Response

```json
{
  "status": "ok",
  "processed": 150,
  "succeeded": 150,
  "failed": 0,
  "message": "Bulk operation completed"
}
```

## Delete All Jobs

```http
DELETE /api/v1/jobs?confirm=true
```

### Query Parameters

| Parameter | Type    | Required | Description                           |
| --------- | ------- | -------- | ------------------------------------- |
| `confirm` | string  | Yes      | Must be `"true"` to confirm deletion  |

### Response

```json
{
  "status": "ok",
  "processed": 500,
  "succeeded": 500,
  "failed": 0,
  "message": "Bulk operation completed"
}
```

<Aside type="danger">
  This operation cannot be undone. All jobs including pending and running jobs will be deleted.
</Aside>

## Job Types

| Type      | Description                                      |
| --------- | ------------------------------------------------ |
| `index`   | Index a repository for searching                 |
| `sync`    | Sync repositories from a code host connection    |
| `replace` | Find-and-replace operation across repositories   |

## Job Payloads

Each job type has a specific payload structure:

### Index Payload
```json
{
  "repository_id": 1,
  "connection_id": 1,
  "repo_name": "myorg/api",
  "clone_url": "https://github.com/myorg/api.git",
  "branch": "main"
}
```

### Sync Payload
```json
{
  "connection_id": 1
}
```

### Replace Payload
```json
{
  "search_pattern": "oldFunction",
  "replace_with": "newFunction",
  "is_regex": false,
  "case_sensitive": true,
  "file_patterns": ["*.go", "*.ts"],
  "matches": [
    {"repository_id": 1, "repository_name": "myorg/api", "file_path": "internal/service.go"},
    {"repository_id": 1, "repository_name": "myorg/api", "file_path": "internal/handler.go"}
  ],
  "branch_name": "refactor/rename-function",
  "mr_title": "Rename oldFunction to newFunction",
  "mr_description": "Automated refactoring"
}
```

<Aside type="note">
  The `matches` array is populated from the preview response. A merge request is always created - changes are never committed directly to main. See [Replace API](/api/replace/) for the two-phase flow.
</Aside>

## Status Values

| Status      | Description                     |
| ----------- | ------------------------------- |
| `pending`   | Waiting in queue                |
| `running`   | Currently executing             |
| `completed` | Finished successfully           |
| `failed`    | Error occurred (check `error`)  |

<Aside>
  Note: There is no `cancelled` status. Cancelled jobs are marked as `failed` with a cancellation message.
</Aside>
